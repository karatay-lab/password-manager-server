# AI Analysis: v1 Plan

This file is my reading of `plan.md` — gaps, ambiguities, and a structured implementation roadmap.

---

## 1. Critical Clarifications

### Encryption flow is underspecified
The plan says "decrypt with private key of server" but Ed25519 is a **signing** scheme, not an encryption scheme. Ed25519 cannot encrypt/decrypt payloads. The intended flow is likely:

| Step | What actually happens | Primitive |
|---|---|---|
| Client → Server | Client encrypts password with `serverPublicKey` | Needs an asymmetric encryption scheme (RSA or X25519 ECDH), NOT Ed25519 |
| Server → DB | Server decrypts, re-encrypts with `DATABASEENCRPYTSECRET` | AES-256-GCM |
| Server → Client | Server decrypts from DB, encrypts with `clientPublicKey` | Needs asymmetric encryption |
| Client decrypts | Uses its own private key | Asymmetric decryption |

**Recommendation**: Use `x25519-dalek` for ECDH key exchange to derive a shared secret, then use AES-256-GCM with that shared key. Ed25519 remains for signatures/authentication only.

Alternatively, use `rsa` crate for RSA-OAEP encryption. But the cleanest approach: **X25519 ECDH + AES-256-GCM** for the asymmetric part, Ed25519 for signing tokens/challenges.

### `/greet` returns `serverPublicKey` but client also sends its public key
The plan says client sends `pub` (its public key) and server returns `[PUBLICKEY]` — is that the server's public key? Makes sense: client and server exchange public keys at `/greet`, then use ECDH to derive shared secrets per session.

### Device token initial state
At `/greet`, `deviceToken` is left null. At `/register`, client sends `token` encrypted with the public key. The plan should clarify: does the client generate the device token, or does the server generate it? Based on the flow, **client generates** the device token and sends it encrypted.

### `/register` needs to store more data
The plan returns 200 but doesn't say it stores `deviceToken`, `ehloSecret` (decrypted), and sets `isConfirmed = false`. These should be persisted to the identity row.

### `/refresh` payload confusion
Plan says payload has `token: [DVC]` but also says `identity.update(token)`. Then `/verify` uses `deviceToken` as a header. Need to align naming — is it `token` or `deviceToken`?

### `/pwd/list` sort parameter
`sort?: Vec<Vec<String, "asc" | "desc">>` is non-trivial to implement with Diesel. Recommend simplifying to a single field + direction, or implementing with raw SQL.

### `validSinceDays` semantics
How is expiration computed? If `validSinceDays = 30` and `createdAt = June 1`, does it expire on July 1? The query `Days(now() - createdAt) < validSinceDays` means the password is valid for `validSinceDays` from creation, then expires. This is clear — good.

## 4. Rust-Specific Concerns

### Diesel + SQLite are synchronous
Diesel uses blocking I/O. From Axum/Tokio handlers, wrap all Diesel calls in `tokio::task::spawn_blocking`. Or use a dedicated `r2d2` pool managed outside the async runtime. The pool should be wrapped in `Arc<Pool<SqliteConnection>>`.

### `edition = "2024"` implications
- `std::env::set_var` is `unsafe` → must call `dotenvy::dotenv()` at the very top of `main`, before `#[tokio::main]` spawns threads
- New error-in-core changes may affect `thiserror`/`anyhow` behavior — verify compatibility
- `gen` is a reserved keyword — avoid as identifier

### SQLite limitations
- No `enum` type — use `TEXT CHECK(...)` or application-level validation
- `BOOLEAN` is stored as `0`/`1` integers
- `BLOB` for key bytes is fine
- `VARCHAR(15)` for IPv4 only — IPv6 needs `VARCHAR(45)`
- No native UUID type — store as `TEXT`
- Concurrent writes are serialized by SQLite's writer lock — fine for single-server deployment

### Argon2 + `spawn_blocking`
Argon2 is deliberately slow (memory-hard). Running it on the async runtime would stall other tasks. Always wrap Argon2 hashing/verification in `tokio::task::spawn_blocking`.

## 5. Endpoint Data Flow Diagrams

### POST /greet
```
Client → Server: { ip, clientPublicKey }
  Server:
    1. Check IP not already registered (412 if exists)
    2. Check clientPublicKey not already registered (409 if exists)
    3. Generate Ed25519 keypair (serverPrivateKey, serverPublicKey)
    4. INSERT identity (ip, server keys, client key, isConfirmed=false)
    5. Respond: { serverPublicKey }
```

### POST /register
```
Client → Server: { ip, token(encrypted with serverPublicKey), ehlo(encrypted with serverPublicKey) }
  Server:
    1. Find identity by IP
    2. Decrypt token + ehlo with serverPrivateKey
    3. Store deviceToken + ehloSecret on identity
    4. Respond: 200
```

### POST /pwd/create
```
Client → Server: { deviceToken, ip, pwd(encrypted with serverPublicKey), groupID }
  Server:
    1. Validate deviceToken + IP match session
    2. Decrypt pwd with serverPrivateKey
    3. Re-encrypt pwd with DATABASEENCRPYTSECRET (AES-256-GCM)
    4. INSERT password row with encrypted pwd
    5. Read back, decrypt with AES-256-GCM
    6. Re-encrypt with clientPublicKey
    7. Respond: { pwd: clientEncryptedPwd, createdAt }
```

### GET /pwd/get/[uuid]
```
Client → Server: { deviceToken, ip, uuid }
  Server:
    1. Validate deviceToken + IP match session
    2. Find password by uuid + identity's group
    3. Fetch group details (name, extra)
    4. Decrypt pwd with DATABASEENCRPYTSECRET
    5. Re-encrypt with clientPublicKey
    6. Respond: { pwd, expires, createdAt, validSinceDays, group: { name, extra } }
```

## 6. Questions to Resolve

1. **Ed25519 can't encrypt** — what asymmetric encryption scheme instead? (X25519 ECDH recommended)
2. **Device token**: client-generated or server-generated?
3. **`/refresh`** — does it need `ehlo` to authorize the token rotation, or just the old token?
4. **`/re-sign` ehlo verification** — decrypt `ehlo` with `serverPrivateKey`, but how to verify it's correct? Compare against stored `ehloSecret`?
5. **Group ownership** — can a password be created without a group? Plan says `groupID: UUID | null` but schema has `NOT NULL` FK.
6. **Pagination** — `take`=page number or `take`=offset? `size`=page size? Need to define.
7. **Search** — what fields does `search` query against? Name? Extra? Both?
8. **Export/import format** — just the SQLite file? Or JSON dump? rsync of the file is simplest.
9. **Admin approval** — how does the admin set `isConfirmed = true`? No endpoint for this in the plan.
10. **CORS** — what origins are allowed? Development-only or production config?
